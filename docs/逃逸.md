# 宿主机安全加固清单

> 在 PVM + Cloud Hypervisor 环境下，对宿主机进行的安全加固及验证结果。

---

## 测试环境

| 项目 | 值 |
|------|-----|
| 宿主机 | 120.24.111.246（OpenCloudOS 9 + PVM） |
| VMM | Cloud Hypervisor v52.0（官方原版） |
| Guest | Debian 12（debootstrap minbase） |
| 隔离方式 | 硬件虚拟化（PVM shadow page table） |

## 加固项

| 优先级 | 操作 | 命令 | 状态 |
| --- | --- | --- | --- |
| 🔴 P0 | 隐藏内核符号 | `echo 2 > /proc/sys/kernel/kptr_restrict` | ✅ 已执行 |
| 🔴 P0 | 禁用 io_uring | `echo 2 > /proc/sys/kernel/io_uring_disabled` | ✅ 已执行 |
| 🔴 P0 | 限制 /dev/mem | 重新编译内核 `CONFIG_STRICT_DEVMEM=y` | ⏸️ 暂缓（需重编内核） |
| 🟡 P1 | 限制用户命名空间 | `echo 0 > /proc/sys/user/max_user_namespaces` | 待定 |
| 🟡 P1 | 限制 FUSE 权限 | `chmod 600 /dev/fuse` | 待定 |
| 🟡 P1 | 限制 TUN 设备 | `chmod 600 /dev/net/tun` | 待定 |
| 🟢 P2 | 启用 SELinux | 不采纳使用 SELinux enforcing 模式，太麻烦了 | 不采纳 |

## 逃逸测试

从 VM 内部尝试各种逃逸向量：

### 测试结果总览

| 逃逸向量 | 结果 | 风险等级 |
|----------|------|---------|
| 读取宿主内存 (/dev/mem) | ❌ 只能读到 VM 自己的内存 | ✅ 安全 |
| 写入宿主内存 | ❌ 写入失败 | ✅ 安全 |
| 访问 /dev/kvm | ❌ 不存在 | ✅ 安全 |
| 访问宿主磁盘 | ❌ 只有 VM 自己的 /dev/vda（5GB） | ✅ 安全 |
| VFIO 设备直通 | ⚠️ /dev/vfio 存在但无实际设备 | ✅ 安全 |
| virtiofs/9p 共享文件系统 | ❌ 未配置 | ✅ 安全 |
| vsock 通信 | ⚠️ /dev/vsock 存在但无宿主监听 | ✅ 安全 |
| 网络逃逸 | ❌ 只能通过 NAT 访问外网 | ✅ 安全 |
| 加载内核模块 | ⚠️ modules_disabled=0（VM 内核配置） | 低风险 |
| seccomp | ❌ 未启用 | 低风险 |
| user namespace | ✅ 可创建 | 低风险 |

### 详细测试记录

#### 1. 内存隔离

```
VM 内 /dev/mem 只能读到 VM 自己的物理内存，包含 CH 的 ACPI 表（"CLOUDH" 字符串）。
无法读取宿主内存。写入 /dev/mem 失败。
```

#### 2. 设备隔离

```
PCI 设备：
  0x8086  Intel host bridge
  0x1af4  virtio-blk（磁盘）
  0x1af4  virtio-net（网络）
  0x1af4  virtio-vsock

无 VFIO 直通设备，无 GPU 直通，无宿主设备暴露。
```

#### 3. 文件系统隔离

```
VM 只能看到自己的 /dev/vda（5GB qcow2）。
无 9p/virtiofs 共享文件系统。
无法挂载宿主任何设备。
```

#### 4. 网络隔离

```
VM 网络：10.100.0.2/24，网关 10.100.0.1
只能通过 NAT 访问外网，无法直接访问宿主内网。
无法嗅探其他 VM 的流量（独立 TAP 设备）。
```

#### 5. 进程隔离

```
VM 有完全独立的进程空间。
ps aux 只能看到 VM 自己的进程。
看不到宿主的 cloud-hypervisor 或其他进程。
```

## 与 LXC 容器对比

| 维度 | Cloud Hypervisor (PVM) | LXC 容器 |
|------|----------------------|----------|
| **隔离级别** | 硬件虚拟化（完整 VM） | OS 级容器（共享内核） |
| **内核** | 独立内核，每个 VM 自己的 | 共享宿主内核 |
| **进程隔离** | 完全独立进程空间，宿主不可见 | namespace 隔离，宿主 root 可见 |
| **文件系统** | 独立磁盘镜像（qcow2） | 共享宿主文件系统（bind mount） |
| **网络** | 独立 TCP/IP 栈，独立 TAP | 共享宿主网络栈（veth + namespace） |
| **设备访问** | 仅 virtio 模拟设备 | 可直接访问宿主设备（/dev/*） |
| **内存隔离** | 硬件页表隔离，VM 物理内存独立 | cgroup 限制，但共享内核内存 |
| **逃逸难度** | 需要 hypervisor 漏洞（极难） | 内核漏洞即可逃逸（较容易） |
| **内核漏洞影响** | VM 内核漏洞只影响该 VM | 宿主内核漏洞影响所有容器 |
| **性能开销** | 有一定开销（PVM 约 10-30%） | 接近原生（<5%） |
| **内存开销** | 每 VM 额外 ~40MB（CH 进程） | 极低（共享内核） |
| **启动速度** | 3-5 秒（内核引导） | <1 秒 |
| **镜像大小** | 需要完整 rootfs（数百 MB） | 共享宿主库，很小 |
| **适用场景** | 不信任的用户、多租户隔离 | 信任的环境、微服务部署 |

### 逃逸风险对比

| 攻击向量 | Cloud Hypervisor | LXC |
|----------|-----------------|-----|
| 内核漏洞提权 | ❌ VM 内核漏洞无法逃逸到宿主 | ✅ 宿主内核漏洞可逃逸 |
| 容器逃逸（namespace/cgroup） | ❌ 不适用，无 namespace | ✅ 配置不当可逃逸 |
| 共享文件系统逃逸 | ❌ 无共享文件系统 | ✅ bind mount 配置不当可逃逸 |
| 设备访问逃逸 | ❌ 仅 virtio 模拟设备 | ✅ 暴露 /dev/* 可逃逸 |
| 网络嗅探 | ❌ 独立 TAP，无法嗅探其他 VM | ⚠️ 同 bridge 下可能嗅探 |
| /proc /sys 信息泄露 | ❌ 只看到 VM 自己的 | ⚠️ 配置不当可泄露宿主信息 |
| virtio/设备模拟漏洞 | ⚠️ 理论风险（需 CH 0day） | ❌ 不适用 |

### 结论

| 场景 | 推荐 |
|------|------|
| 给不信任的用户切 VPS | ✅ Cloud Hypervisor（完整 VM 隔离） |
| 给朋友临时用 | ✅ Cloud Hypervisor（省心） |
| 自己的服务部署 | ✅ LXC（性能好、轻量） |
| 多租户生产环境 | ✅ Cloud Hypervisor |
| 单机高密度部署 | ⚠️ LXC 更省资源，但需信任用户 |

---

## 结论

**CH 提供完整 VM 隔离（硬件虚拟化），目前未发现可行的逃逸路径。**

潜在攻击面（均为理论风险，需要具体 CVE）：

| 攻击面 | 说明 | 缓解 |
|--------|------|------|
| virtio 设备实现 | CH 的 virtio-blk/net/vsock 实现可能存在漏洞 | 保持 CH 版本更新 |
| PVM 内核模块 | kvm_pvm 模块可能存在漏洞 | 保持 PVM 内核更新 |
| Guest 内核漏洞 | VM 内核漏洞可用于提权，但无法逃逸到宿主 | 定期更新 guest 内核 |
